# Finance Bot: текущая логика

Актуально для версии с раздельными чатами согласования и оплаты.

## Назначение

Бот принимает счета через личный диалог или Telegram Mini App и отправляет их на согласование. Обычные счета после согласования публикуются в отдельном чате оплаты в назначенную дату. Счета категории `Такси` завершаются после согласования и в чат оплаты не поступают.

## Основной маршрут счета

1. Автор создает счет через `/new` или Mini App.
2. Автор указывает проект, статью расхода, получателя платежа, сумму, файл (необязательно) и комментарий. Дата оплаты запрашивается для всех статей, кроме `Такси`.
3. Бот создает строку в листе `requests` со статусом `На согласовании`.
4. Бот отправляет счет только в `approval_chat_id` выбранного проекта.
5. В чате согласования доступны кнопки `Одобрить` и `Отклонить`.
6. После одобрения сообщение остается в чате согласования, меняется на `Счет #N — Согласован✅`, кнопки исчезают.
7. До даты оплаты бот ничего не отправляет в чат оплаты.
8. В дату отправки бот один раз публикует счет в `payment_chat_id` с кнопками `Оплатил – прикрепить чек` и `Отменить счет`.
9. После загрузки чека счет меняется на `Счет #N — Оплачен✅`, сообщение пользователя с чеком удаляется, а бот публикует чек ответом на сообщение счета.
10. Автор счета получает чек и кнопки `Да` / `Нет` под вопросом `Оплата получена?`.

Для `Такси` маршрут заканчивается на шаге 6: после согласования автор получает итоговое сообщение, а отправка на оплату и работа с чеком не запускаются.

## Дата оплаты

Дата обязательна для всех новых счетов, кроме категории `Такси`. В Mini App поле даты скрывается после выбора этой категории; в диалоге бот сразу переходит к загрузке файла.

В личном диалоге поддерживаются форматы:

- `15` — ближайшее подходящее 15-е число;
- `15.07` — ближайшая такая дата, при необходимости в следующем году;
- `15.07.2026`;
- `2026-07-15`.

Mini App использует календарь и отправляет дату в формате `YYYY-MM-DD`.

Полная дата в прошлом отклоняется. Короткая дата без года переносится на следующий год, если в текущем году уже прошла. День без месяца выбирает ближайший будущий месяц, в котором такой день существует.

## Когда счет появляется в чате оплаты

Фоновая задача проверяет счета каждые 5 минут.

- Будний день: счет отправляется с 10:00 по часовому поясу `REMINDER_TIMEZONE`.
- Суббота: счет отправляется в предшествующую пятницу с 10:00.
- Воскресенье: счет отправляется в предшествующую пятницу с 10:00.
- Если бот был выключен в нужное время, просроченный счет отправляется при следующей успешной проверке.
- Если счет согласовали после наступления времени отправки, бот отправляет его в чат оплаты сразу после согласования.
- Если счет согласовали раньше, он ждет рассчитанной даты отправки.
- Повторной ежедневной или понедельничной рассылки нет.

Счет отправляется только при одновременном выполнении условий:

- статус строки равен `Согласован`;
- заполнена корректная `payment_due_date`;
- поле `payment_message_id` пустое;
- наступили рассчитанные дата и время.

Старые строки без `payment_due_date` автоматически не рассылаются. Это защищает рабочие чаты от массовой отправки исторических счетов.

Категория `Такси` дополнительно исключается из этого планировщика независимо от статуса и даты.

## Согласование

Технического ограничения по Telegram ID согласующего нет. Нажать кнопку может участник чата, которому доступно сообщение.

Кнопка срабатывает только если:

- строка имеет статус `На согласовании`;
- Telegram chat ID совпадает с `approval_message_chat_id`;
- Telegram message ID совпадает с `approval_message_id`.

Это блокирует действия со старых или дублированных сообщений.

### Одобрить

- статус меняется на `Согласован`;
- сохраняются время и имя согласующего;
- сообщение в чате согласования редактируется, а кнопки удаляются;
- автор получает сообщение `Ваш счет согласован` с названием и датой оплаты;
- счет отправляется в чат оплаты только если его расчетное время уже наступило.

### Отклонить

- бот просит причину;
- после ввода причины статус меняется на `Отклонен`;
- исходное сообщение не удаляется: оно меняется на `Отклонен❌`, содержит причину и остается в истории;
- кнопки исчезают;
- автор получает причину и просьбу отправить исправленный счет заново.

Напоминаний о несогласованных счетах нет. Наличие кнопок в чате согласования означает, что счет еще ожидает решения.

## Оплата

Кнопки оплаты работают только для строки со статусом `Согласован` и только на текущем сообщении, указанном в `payment_chat_id` + `payment_message_id`.

### Оплатил – прикрепить чек

- бот просит файл чека;
- перед обработкой файла повторно проверяет статус и ID текущего сообщения;
- статус, оплативший, Telegram file ID и тип файла сохраняются одним обновлением строки;
- сообщение счета редактируется на `Оплачен✅`, кнопки исчезают;
- пользовательское сообщение с чеком удаляется;
- бот повторно публикует чек ответом на счет;
- автору отправляется чек для подтверждения получения оплаты.

Если Telegram не смог изменить счет или повторно опубликовать чек, бот возвращает статус
`Согласован`, восстанавливает кнопки оплаты и не удаляет сообщение пользователя с чеком.

### Отменить счет

- бот просит причину;
- статус меняется на `Отменен`;
- сообщение в чате оплаты меняется на `Отменен❌`, причина остается в тексте;
- кнопки исчезают;
- автор получает уведомление.

## Подтверждение автора

### Да

Кнопки подтверждения исчезают. Бот отправляет автору стандартное сообщение `Напиши /new чтобы отправить счет`.

### Нет

- статус возвращается из `Оплачено` в `Согласован`;
- чек отправляется обратно в чат оплаты с тегом оплатившего и текстом `Оплата по данному чеку не получена`;
- это новое сообщение становится текущим платежным сообщением;
- под ним снова доступны кнопки оплаты и отмены.

## Статьи расходов

Доступны четыре значения:

- `Команда`;
- `Рекламный бюджет`;
- `Сервисы`;
- `Такси`.

Для `Такси` используется пример комментария:

*Пример*

??? сом - итоговая сумма за такси
Цель поездки: ???

После согласования обычного такси-счета:

- статус остается `Согласован`;
- автор получает получателя и комментарий;
- счет не отправляется в чат оплаты;
- напоминания, кнопка оплаты и чек для него не создаются.

### Сводные счета за такси

Фоновая задача проверяет необходимость сводных счетов каждые 5 минут и начинает работу с 10:00 по `REMINDER_TIMEZONE`:

- 5-го числа учитываются согласованные такси-счета, созданные с 16-го числа предыдущего месяца по последнее число предыдущего месяца включительно;
- 20-го числа учитываются согласованные такси-счета, созданные с 1-го по 15-е число текущего месяца включительно;
- суммы группируются по автору и проекту;
- для каждой группы создается один сводный счет со статусом `На согласовании`;
- сводный счет содержит итоговую сумму, точный период и Telegram-тег автора;
- после согласования сводный счет отправляется автору и также больше никуда не пересылается.

В расчет входят только строки со статусом `Согласован`. Отклоненные, отмененные, оплаченные и еще не согласованные строки не учитываются. Если хотя бы одна сумма группы не распознана, сводный счет для этой группы не создается, а ошибка записывается в логи.

Уникальный ключ периода, проекта и автора сохраняется в колонке `T`. Поэтому повторные проверки в течение 5-го или 20-го числа не создают дубликаты. Если строка сводного счета записалась, но сообщение не отправилось, планировщик повторяет только отправку этого сообщения.

Статья сохраняется в строке счета. Старая понедельничная группировка больше не используется.

Особое правило: для проектов `ОР`, `ОР КГ`, `ОРКГ` и статьи `Рекламный бюджет` плательщик принудительно меняется на `@bulat_sufyanov`. Для остальных статей используется `payer_tag` из листа `projects`.

## Лист projects

Столбцы:

| Столбец | Поле | Назначение |
|---|---|---|
| A | `project` | Аббревиатура или алиас проекта |
| B | `payment_chat_id` | Чат, куда счет попадет в дату оплаты |
| C | `payer_tag` | Кого тегать в чате оплаты |
| D | `approval_chat_id` | Чат первоначального согласования |

Для каждого используемого алиаса должен быть заполнен `approval_chat_id`. Для обычных счетов также обязателен `payment_chat_id`; для категории `Такси` он не используется.

Допускается одинаковый ID в B и D, но тогда согласование и оплата фактически происходят в одном чате.

## Статусы и допустимые переходы

```text
На согласовании -> Согласован -> Оплачено
На согласовании -> Отклонен
Согласован -> Отменен
Оплачено -> Согласован  (автор нажал «Нет»)
Согласован -> Оплачено  (загружен новый чек)
На согласовании -> Согласован  (Такси: финальный переход без оплаты)
```

Любая кнопка, не соответствующая текущему статусу или текущему Telegram message ID, отвечает `Это сообщение уже неактуально` и ничего не меняет.

## Переменные Render

Обязательные:

- `BOT_TOKEN` — токен рабочего бота;
- `GOOGLE_CREDENTIALS` — JSON сервисного аккаунта Google;
- `WEBAPP_URL` — HTTPS URL Mini App, обычно URL Render + `/miniapp`.

Настройки расписания:

- `REMINDER_TIMEZONE` — часовой пояс, по умолчанию `Asia/Bishkek`;
- `PAYMENT_DISPATCH_HOUR` — час отправки, по умолчанию `10`;
- `PAYMENT_DISPATCH_MINUTE` — минуты, по умолчанию `0`;
- `PAYMENT_DISPATCH_INTERVAL_SECONDS` — частота проверки, по умолчанию `300`.

Для обратной совместимости час и минуты также могут подхватиться из старых `REMINDER_HOUR` и `REMINDER_MINUTE`.

Служебные переменные миграции (`NEW_BOT_TOKEN`, `MIGRATION_SECRET`) относятся к разовому сценарию смены бота и не участвуют в обычном создании счетов.

## HTTP и Render

- `GET /` и `GET /health` возвращают `200 OK`.
- `HEAD /` и `HEAD /health` возвращают `200`, поэтому UptimeRobot поддерживает Render активным без ошибок `501`.
- `GET /miniapp` отдает интерфейс Mini App.
- `POST /api/requests` создает счет из Mini App.

## Правила безопасных изменений

1. Не менять индексы колонок без одновременного изменения заголовков и всех чтений/записей.
2. Не использовать `approval_message_id` как ID сообщения оплаты: это разные сообщения.
3. Не очищать `payment_message_id` без причины, иначе планировщик может отправить дубликат.
4. Обычную отправку на оплату ограничивать статусом, заполненной `payment_due_date` и исключением категории `Такси`.
5. После изменения кнопок проверять защиту от старых сообщений.
6. Перед деплоем запускать `python -m unittest -v test_payment_schedule.py test_taxi_reimbursements.py` и `python -m py_compile bot.py payment_schedule.py taxi_reimbursements.py migrate_active_invoices.py`.
7. Перед включением новых проектов проверять заполнение `payment_chat_id` и `approval_chat_id` для каждого алиаса.
