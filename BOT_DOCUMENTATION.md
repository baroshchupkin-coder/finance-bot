# Документация по Telegram finance bot

Актуально для кода в `bot.py` и `migrate_active_invoices.py` на 2026-07-08.

Документ описывает текущую бизнес-логику бота: состояния счета, кнопки, зависимости от Google Sheets, напоминания, Mini App и миграцию на нового бота.

## 1. Что делает бот

Бот автоматизирует оплату счетов в Telegram:

1. Пользователь создает счет в личном чате через `/new` или через Telegram Mini App.
2. Бот записывает счет в Google Sheets.
3. Бот отправляет счет на согласование в чат проекта.
4. Согласующий одобряет или отклоняет счет.
5. Если счет одобрен, бот отправляет его в рабочий чат с кнопками оплаты.
6. Оплатчик нажимает `Оплатил - прикрепить чек` и прикладывает чек или платежное поручение.
7. Бот отправляет чек автору счета и спрашивает, получена ли оплата.
8. Если автор подтверждает получение, процесс завершается.
9. Если автор не подтверждает получение, бот возвращает чек в рабочий чат и снова оставляет счет в статусе `Согласован`.
10. Если счет зависает, бот напоминает о нем на следующий день и далее по понедельникам.

## 2. Основные файлы

| Файл | За что отвечает |
| --- | --- |
| `bot.py` | Основной Telegram-бот, Google Sheets, напоминания, Mini App API, web health endpoint, browser migration endpoint. |
| `miniapp.html` | HTML-форма Telegram Mini App для создания счета. |
| `migrate_active_invoices.py` | Разовый перенос активных счетов со старого Telegram-бота на нового. |
| `MIGRATION.md` | Инструкция по миграции активных счетов на нового бота. |
| `render.yaml` | Конфигурация сервиса Render. |
| `requirements.txt` | Python-зависимости. |

Render запускает `bot.py`. Внутри процесса параллельно работают:

1. Telegram polling через `ApplicationBuilder().run_polling(drop_pending_updates=True)`.
2. HTTP-сервер в отдельном потоке для `/`, `/health`, `/miniapp`, `/api/requests`, `/migration`.
3. Ежедневная задача напоминаний через `JobQueue.run_daily`.

## 4. Переменные окружения

| Переменная | Обязательна | Значение |
| --- | --- | --- |
| `BOT_TOKEN` | Да | Токен текущего Telegram-бота. Все file_id и кнопки привязаны к этому боту. |
| `GOOGLE_CREDENTIALS` | Да | JSON сервисного аккаунта Google для доступа к таблице `Finance bot`. |
| `WEBAPP_URL` | Нет | Полный URL Mini App. Если не задан, но есть `RENDER_EXTERNAL_URL`, бот берет `RENDER_EXTERNAL_URL + /miniapp`. |
| `RENDER_EXTERNAL_URL` | Нет | Render может задавать автоматически. Используется как fallback для `WEBAPP_URL`. |
| `MINIAPP_REQUIRE_INIT_DATA` | Нет | По умолчанию `true`. Если `true`, `/api/requests` принимает заявки только с валидными Telegram Mini App initData. |
| `MINIAPP_DEBUG_USER_ID` | Нет | Используется только если `MINIAPP_REQUIRE_INIT_DATA=false`, для локальных тестов Mini App. |
| `REMINDER_TIMEZONE` | Нет | Часовой пояс напоминаний. По умолчанию `Asia/Bishkek`. |
| `REMINDER_HOUR` | Нет | Час запуска напоминаний. По умолчанию `10`. |
| `REMINDER_MINUTE` | Нет | Минута запуска напоминаний. По умолчанию `0`. |
| `WEEKLY_REMINDER_WEEKDAY` | Нет | День еженедельного напоминания. По умолчанию `0`, то есть понедельник по Python-формату. |
| `REMINDER_EXISTING_ROWS_PAUSE_FROM` | Нет | Дата, до которой старые строки не получают первое ежедневное напоминание. По умолчанию `2026-06-04`. |
| `MIGRATION_SECRET` | Нет | Секрет для запуска `/migration` через браузер. Если не задан, endpoint миграции выключен. |
| `MIGRATION_TIMEOUT_SECONDS` | Нет | Таймаут browser migration endpoint. По умолчанию `600`. |
| `NEW_BOT_TOKEN` | Только для миграции | Токен нового Telegram-бота при переотправке активных счетов. |
| `MIGRATION_SPREADSHEET_NAME` | Нет | Имя таблицы для миграции. По умолчанию `Finance bot`. |
| `MIGRATION_REQUESTS_SHEET_NAME` | Нет | Лист заявок для миграции. По умолчанию `requests`. |
| `MIGRATION_LOGS_SHEET_NAME` | Нет | Лист логов миграции. По умолчанию `logs`. |

Если оставить дефолтный `REMINDER_TIMEZONE=Asia/Bishkek` и `REMINDER_HOUR=10`, напоминания идут в 10:00 по Бишкеку. Для Новосибирска это 11:00.

## 5. Google Sheets

Бот открывает таблицу `Finance bot` и использует листы:

| Лист | Использование |
| --- | --- |
| `requests` | Основная таблица счетов. |
| `projects` | Справочник проектов: куда отправлять счет и кого тегать на оплату. |
| `logs` | Логи миграции активных счетов. Создается или используется скриптом миграции. |

### 5.1. Лист `projects`

Код ожидает такие колонки:

| Индекс | Смысл |
| --- | --- |
| 0 | Аббревиатура или название проекта, которое пользователь вводит в боте. Сравнение без учета регистра и пробелов по краям. |
| 1 | `approver_chat_id` - чат, куда бот отправляет счет на согласование и дальнейшую оплату. |
| 2 | `payer_tag` - тег оплатчика, который показывается в одобренном счете и напоминаниях. |

Если проект не найден в `projects`, бот не создает счет и просит ввести проект заново.

Специальное правило по оплатчику: для проектов `OR`, `ORKG`, `ОР`, `ОР КГ` при статье расхода `Рекламный бюджет` бот подставляет `@bulat_sufyanov` вместо стандартного `payer_tag` из `projects`. Для остальных статей этих проектов остается стандартный тег из таблицы, например `@n0visad`.

### 5.2. Лист `requests`

Колонки в `requests` жестко зашиты в `bot.py`. Нельзя вставлять новые колонки в середину без изменения индексов в коде.

| Индекс | Константа | Что хранится |
| --- | --- | --- |
| 0 | `REQUEST_ID_COL` | Номер счета. Сейчас равен количеству строк в листе на момент создания. |
| 1 | - | Дата создания счета. Для чата берется `update.message.date`, для Mini App - текущее время в `REMINDER_TZ`. |
| 2 | - | Username автора счета. |
| 3 | - | Проект. |
| 4 | - | Кому платим: имя, компания или сервис. |
| 5 | - | Сумма. |
| 6 | - | Комментарий к счету. |
| 7 | `STATUS_COL` | Статус счета. |
| 8 | `APPROVER_CHAT_ID_COL` | Чат согласования и оплаты. |
| 9 | `FILE_ID_COL` | Telegram `file_id` файла или фото счета. |
| 10 | `CREATOR_CHAT_ID_COL` | Telegram ID автора счета. Нужен для личных уведомлений. |
| 11 | - | Имя автора счета. |
| 12 | `APPROVER_NAME_COL` | Username или имя согласующего, который нажал `Одобрить`. |
| 13 | `PAYER_TAG_COL` | Тег оплатчика из листа `projects`. |
| 14 | `APPROVED_AT_COL` | Дата одобрения счета. От нее считается первое напоминание по оплате. |
| 15 | `PAYMENT_CHAT_ID_COL` | Чат, где была нажата кнопка оплаты. |
| 16 | `PAYMENT_PAYER_TAG_COL` | Кто нажал `Оплатил` и приложил чек. |
| 17 | `PAYMENT_RECEIPT_FILE_ID_COL` | `file_id` чека или платежного поручения. |
| 18 | `PAYMENT_RECEIPT_FILE_TYPE_COL` | Тип чека: `photo` или `document`. |
| 19 | `LAST_PAYMENT_REMINDER_AT_COL` | Дата последнего напоминания. Сейчас используется и для согласования, и для оплаты. |
| 20 | `LAST_INVOICE_MESSAGE_CHAT_ID_COL` | Чат последнего отправленного сообщения со счетом. |
| 21 | `LAST_INVOICE_MESSAGE_ID_COL` | Message ID последнего отправленного сообщения со счетом. |
| 22 | `EXPENSE_CATEGORY_COL` | Статья расхода: `Команда`, `Рекламный бюджет`, `Сервисы` или другое значение из старых строк. |

Ключевой инвариант: колонки 20 и 21 всегда указывают на последнее актуальное сообщение со счетом, которое бот может удалить перед повторной отправкой. На этапе согласования это сообщение с кнопками `Одобрить`/`Отклонить`; после одобрения - сообщение с кнопками `Оплатил`/`Отменить счет`.

## 6. Статусы счета

| Статус | Когда ставится | Что означает |
| --- | --- | --- |
| `На согласовании` | При создании счета через `/new` или Mini App. | Счет отправлен ответственному чату, но еще не одобрен. |
| `Согласован` | После кнопки `Одобрить`; также возвращается после ответа `Нет` на вопрос `Оплата получена?`. | Счет можно оплачивать, по нему работают напоминания об оплате. |
| `Оплачено` | После того как оплатчик нажал `Оплатил` и приложил чек. | Чек отправлен автору счета на подтверждение. |
| `Отклонен` | После кнопки `Отклонить` и ввода причины. | Счет закрыт, напоминания по нему не идут. |
| `Отменен` | После кнопки `Отменить счет` на уже согласованном счете и ввода причины. | Счет закрыт, напоминания по нему не идут. |

Напоминания берут только два статуса: `На согласовании` и `Согласован`. Статусы `Оплачено`, `Отклонен`, `Отменен` в напоминания не попадают.
## 7. Создание счета через личный чат

### 7.1. `/start`

Работает только в личном чате.

Если пользователь пишет `/start`, бот отвечает, что можно написать `/new`, чтобы отправить счет. Если настроен `WEBAPP_URL`, дополнительно показывает кнопку `Открыть форму` для Mini App.

Если пользователь пишет обычное сообщение вне сценария создания счета, бот отвечает: `Напиши /new чтобы отправить счет`.

### 7.2. `/new`

Работает только в личном чате. При `/new` бот создает `user_state[chat_id] = {}` и запускает пошаговый сценарий.

### 7.3. Шаг 1 - проект

Бот спрашивает аббревиатуру проекта.

Если проект найден в листе `projects`:

1. В `user_state` сохраняются `project`, `approver_id`, `payer_tag`.
2. Бот спрашивает статью расхода.
3. Показывает кнопки статей расхода.

Если проект не найден:

1. Счет не создается.
2. Бот пишет, что для проекта не найден согласующий.
3. Пользователь остается на шаге ввода проекта.

### 7.4. Шаг 2 - статья расхода

Бот показывает три кнопки:

| Кнопка | Callback | Что сохраняется |
| --- | --- | --- |
| `Команда` | `expense_team` | `Команда` |
| `Рекламный бюджет` | `expense_ads` | `Рекламный бюджет` |
| `Сервисы` | `expense_services` | `Сервисы` |

Если пользователь отправляет текст вместо кнопки, бот просит выбрать статью кнопкой.

После выбора бот редактирует сообщение с выбранной статьей и спрашивает, кому платим.

Важно: статья расхода влияет только на группировку еженедельных понедельничных напоминаний по оплате. В обычном тексте счета она нигде не выводится.

### 7.5. Шаг 3 - кому платим

Пользователь вводит имя, компанию или сервис. Бот сохраняет значение в `state["target"]` и спрашивает сумму.

### 7.6. Шаг 4 - сумма

Пользователь вводит сумму текстом. Бот сохраняет значение в `state["amount"]` и просит прикрепить файл счета или нажать `Пропустить`.

### 7.7. Шаг 5 - файл счета

Если пользователь прикрепил фото или документ:

1. Бот сохраняет Telegram `file_id` в `state["file_id"]`.
2. Ставит `state["file_step_done"] = True`.
3. Просит ввести комментарий.

Если пользователь нажал `Пропустить`:

1. Бот ставит `state["file_step_done"] = True`.
2. Просит ввести комментарий.
3. В колонке `FILE_ID_COL` будет пусто.

Если пользователь отправляет файл не на этом шаге, бот его игнорирует.

### 7.8. Шаг 6 - комментарий

Пользователь вводит комментарий. Бот создает строку в `requests` со статусом `На согласовании`, сохраняет все данные и отправляет пользователю подтверждение: счет принят, ответственный получил уведомление.

Затем бот отправляет счет в `approver_id` из листа `projects`:

- если есть `file_id`, отправляет фото или документ с caption;
- если файла нет, отправляет обычное текстовое сообщение;
- добавляет кнопки `Одобрить` и `Отклонить`;
- сохраняет `chat_id` и `message_id` отправленного сообщения в колонки 20 и 21.

## 8. Создание счета через Mini App

Mini App открывается по `/miniapp` и отправляет форму на `/api/requests`.

Если `MINIAPP_REQUIRE_INIT_DATA=true`, API проверяет Telegram `initData` по `BOT_TOKEN`. Если проверка не прошла, заявка не создается и API возвращает ошибку.

Если `MINIAPP_REQUIRE_INIT_DATA=false`, API использует debug-пользователя и `MINIAPP_DEBUG_USER_ID`. Этот режим нужен только для тестов.

Mini App требует обязательные поля:

- проект;
- статью расхода;
- кому платим;
- сумму;
- комментарий.

Файл не обязателен. Если обязательного поля нет или проект не найден в `projects`, заявка не создается.

После успешной отправки Mini App создает ту же строку в `requests`, что и чатовый сценарий:

1. Статус `На согласовании`.
2. Данные автора из Telegram initData.
3. Проект, сумма, комментарий, статья расхода.
4. Чат согласования и тег оплатчика из `projects`.

Потом API напрямую через Telegram Bot API отправляет счет на согласование, сохраняет `file_id`, `chat_id` и `message_id` в таблицу. Дальнейшая логика полностью общая с обычным счетом из `/new`.

## 9. Согласование счета

### 9.1. Кнопка `Одобрить`

Callback имеет формат `approve_<request_id>`.

Если согласующий нажал `Одобрить`:

1. Бот находит строку по `request_id`.
2. Ставит статус `Согласован`.
3. Записывает дату одобрения в `APPROVED_AT_COL`.
4. Очищает `LAST_PAYMENT_REMINDER_AT_COL`, чтобы первое напоминание по оплате считалось заново от даты одобрения.
5. Записывает username или имя согласующего в `APPROVER_NAME_COL`.
6. Удаляет исходное сообщение со счетом на согласование.
7. Отправляет в тот же чат новое сообщение со счетом и кнопками оплаты.
8. Сохраняет новое `chat_id` и `message_id` в колонки 20 и 21.

Текст одобренного счета содержит тег оплатчика, номер счета, кому платим, комментарий и кто согласовал.

Кнопки после одобрения:

1. `Оплатил - прикрепить чек`.
2. `Отменить счет`.

### 9.2. Кнопка `Отклонить`

Callback имеет формат `reject_<request_id>`.

Если нажали `Отклонить`:

1. Бот просит ввести причину отклонения.
2. В `reject_state[user_id]` сохраняется контекст: номер счета, сообщение, чат, служебное сообщение с просьбой причины.
3. Следующее текстовое сообщение этого пользователя в любом чате считается причиной отклонения.

Когда причина отправлена:

1. Бот удаляет сообщение со счетом, где были кнопки, если может.
2. Бот удаляет сообщение `Введите причину...`, если может.
3. Бот удаляет сообщение пользователя с причиной, если может.
4. Ставит статус `Отклонен`.
5. Отправляет автору счета личное сообщение, что счет не согласован, указывает причину и просит отправить счет заново с учетом комментария.
6. В чат, где была введена причина, отправляет подтверждение, что счет отклонен и комментарий отправлен.

### 9.3. Кнопка `Отменить счет`

Callback имеет формат `cancel_<request_id>`. Эта кнопка есть только на уже согласованном счете.

Механика такая же, как у отклонения:

1. Бот просит ввести причину отмены.
2. Следующее текстовое сообщение пользователя становится причиной.
3. Бот удаляет сообщение со счетом и служебные сообщения, если может.
4. Ставит статус `Отменен`.
5. Отправляет автору счета уведомление, что счет отменен, с причиной.
6. В рабочий чат отправляет подтверждение, что счет отменен и комментарий отправлен.

Отличие от `Отклонить` только в статусе и формулировках.

## 10. Оплата и чек

### 10.1. Кнопка `Оплатил - прикрепить чек`

Callback имеет формат `paid_<request_id>`.

Если оплатчик нажал кнопку:

1. Бот сохраняет в `payment_state[user_id]` номер счета, message ID исходного счета и chat ID рабочего чата.
2. Бот отвечает в рабочем чате: `Прикрепите чек или подтверждение оплаты`.
3. Message ID этого служебного сообщения тоже сохраняется, чтобы потом удалить его.

Следующее фото или документ от этого пользователя считается чеком по этому счету.

### 10.2. Если пользователь приложил чек

Когда пользователь отправляет фото или документ, а его `user_id` есть в `payment_state`:

1. Бот забирает данные из `payment_state` и удаляет это состояние.
2. Определяет `file_id` и тип файла: `photo` или `document`.
3. Находит строку счета в `requests`.
4. Ставит статус `Оплачено`.
5. Записывает рабочий чат в `PAYMENT_CHAT_ID_COL`.
6. Записывает тег оплатчика в `PAYMENT_PAYER_TAG_COL`.
7. Записывает `file_id` чека в `PAYMENT_RECEIPT_FILE_ID_COL`.
8. Записывает тип файла в `PAYMENT_RECEIPT_FILE_TYPE_COL`.
9. Редактирует исходное сообщение со счетом: в начале пишет `Счет #<id> - Оплачен✅`, ниже оставляет данные счета, согласующего и оплатчика.
10. Удаляет сообщение пользователя с чеком, если может.
11. Отправляет чек от имени бота в рабочий чат как reply к исходному счету с подписью `Чек по счету #<id>`.
12. Отправляет чек автору счета в личку с вопросом `Оплата получена?` и кнопками `Да` / `Нет`.
13. Удаляет служебное сообщение `Прикрепите чек...`, если может.

Если Telegram не дает отредактировать caption, бот пробует редактировать текст. Если оба варианта не прошли, ошибка проглатывается и процесс оплаты продолжается.

### 10.3. Сообщение автору счета

Автор получает чек или платежное поручение с подписью:

```text
Счет #<id> по проекту <проект> оплачен

Сумма: <сумма>

Оплата получена?
```

Под сообщением две кнопки:

1. `Да` - callback `received_yes_<request_id>`.
2. `Нет` - callback `received_no_<request_id>`.

### 10.4. Если автор нажал `Да`

Если автор нажал `Да`:

1. Бот убирает кнопки `Да` / `Нет` с сообщения в личке.
2. Статус остается `Оплачено`.
3. Бот пишет автору: `Напиши /new чтобы отправить счет`.
4. Никаких новых сообщений в рабочий чат не отправляется.

### 10.5. Если автор нажал `Нет`

Если автор нажал `Нет`:

1. Бот убирает кнопки `Да` / `Нет` с сообщения в личке.
2. Находит данные чека в `requests`.
3. Ставит статус обратно в `Согласован`.
4. Записывает сегодняшнюю дату в `LAST_PAYMENT_REMINDER_AT_COL`, чтобы счет не попал в повторное напоминание сразу в тот же день.
5. Отправляет чек обратно в рабочий чат.
6. В подписи тегает оплатчика и пишет: `Оплата по данному чеку не получена`.
7. Под возвращенным чеком снова есть кнопка `Оплатил - прикрепить чек`, чтобы можно было приложить корректный чек.

Если в таблице нет `payment_chat_id` или `receipt_file_id`, бот сообщает автору, что не удалось вернуть чек.
## 11. Напоминания

Напоминания запускаются один раз в день через `JobQueue.run_daily`.

Время задается так:

```text
REMINDER_TIMEZONE, REMINDER_HOUR, REMINDER_MINUTE
```

По умолчанию это 10:00 в `Asia/Bishkek`.

### 11.1. Какие счета попадают в напоминания

Бот проходит все строки `requests`, кроме заголовка, и для каждой строки вычисляет `reminder_kind`.

В напоминания попадают только:

1. `На согласовании` - если нужно напомнить о согласовании.
2. `Согласован` - если нужно напомнить об оплате.

Не попадают:

- `Оплачено`;
- `Отклонен`;
- `Отменен`;
- строки без подходящей даты;
- строки, по которым напоминание уже отправлялось сегодня или после последнего понедельника.

### 11.2. Первое напоминание

Если по счету еще не было напоминаний (`LAST_PAYMENT_REMINDER_AT_COL` пустая):

- для статуса `На согласовании` бот смотрит на дату создания счета;
- для статуса `Согласован` бот смотрит на дату одобрения `APPROVED_AT_COL`.

Если эта дата раньше сегодняшнего дня, бот отправляет первое напоминание.

Пример для согласования:

1. Счет создан 04.06 и остался `На согласовании`.
2. 05.06 во время job бот отправит напоминание на согласование.

Пример для оплаты:

1. Счет одобрен 04.06 и остался `Согласован`.
2. 05.06 во время job бот отправит напоминание об оплате.

### 11.3. Старые строки до `REMINDER_EXISTING_ROWS_PAUSE_FROM`

Для строк, дата которых раньше `REMINDER_EXISTING_ROWS_PAUSE_FROM`, первое ежедневное напоминание отключено.

По умолчанию граница `2026-06-04`.

Зачем это нужно: чтобы старые тестовые и давно оплаченные счета не начали массово дублироваться на следующий день после внедрения напоминаний.

Такие строки могут попасть только в еженедельное понедельничное напоминание, если подходят по статусу.

### 11.4. Повторные напоминания

Если в `LAST_PAYMENT_REMINDER_AT_COL` уже есть дата:

1. Бот не отправляет напоминание каждый день.
2. Бот ждет день недели `WEEKLY_REMINDER_WEEKDAY`.
3. По умолчанию это понедельник.
4. Если сегодня понедельник и последнее напоминание было раньше сегодняшнего дня, бот снова отправит счет.

То есть механика такая:

- первое напоминание - на следующий день после создания или одобрения;
- дальше - только по понедельникам.

### 11.5. Куда отправляется напоминание

Напоминания группируются по `APPROVER_CHAT_ID_COL`.

Это значит, что и несогласованные счета, и согласованные неоплаченные счета отправляются в тот чат, который указан для проекта в листе `projects`.

### 11.6. Вводный текст напоминания

Перед счетами бот отправляет одно текстовое сообщение.

Если есть неоплаченные согласованные счета и у всех один `payer_tag`, бот начинает с тега оплатчика:

```text
@payer, напоминаю про оплаты.
```

Если оплатчиков несколько или тега нет:

```text
Напоминаю про оплаты.
```

Если в напоминании только счета на согласование:

```text
Напоминаю про счета на согласование.
```

Если есть счета на согласование, бот добавляет фразу, что сначала дублирует счета, которые ждут согласования.

Если есть счета к оплате, бот добавляет инструкцию: если счета были оплачены, нужно нажать кнопку `оплатил` и прикрепить платежные поручения.

### 11.7. Порядок блоков в напоминании

Порядок всегда такой:

1. Общий вводный текст.
2. Если есть несогласованные счета: заголовок `На согласование` жирным.
3. Все счета в статусе `На согласовании`, у которых пришло время напоминания.
4. Все счета в статусе `Согласован`, у которых это первое напоминание после одобрения. Они идут без группировки по статьям расхода.
5. Все счета в статусе `Согласован`, у которых это еженедельное понедельничное напоминание. Они группируются по статьям расхода.

Для понедельничных оплат порядок категорий такой:

1. `Команда`.
2. `Рекламный бюджет`.
3. `Сервисы`.
4. Остальные категории, если есть старые или нестандартные значения.

Важно: группировка по статьям расходов работает только для еженедельного понедельничного блока оплаты. Первое напоминание на следующий день не группируется по статьям.

### 11.8. Удаление старых дублей при напоминании

Перед тем как переотправить счет в напоминании, бот вызывает `delete_last_invoice_message`.

Он берет из строки:

- `LAST_INVOICE_MESSAGE_CHAT_ID_COL`;
- `LAST_INVOICE_MESSAGE_ID_COL`.

Если Telegram разрешает удалить старое сообщение, бот удаляет его и отправляет новый дубль ниже по чату.

После отправки бот сохраняет новое сообщение в колонки 20 и 21. Это предотвращает накопление одинаковых счетов в чате.

Ограничения:

- Telegram может не дать удалить слишком старое сообщение.
- У бота должны быть права на удаление сообщений в группе.
- Старые сообщения, которые были отправлены до появления колонок 20 и 21 или до сохранения message_id, бот удалить не сможет.

### 11.9. Что меняется в таблице после напоминания

Для каждого переотправленного счета бот:

1. Записывает сегодняшнюю дату в `LAST_PAYMENT_REMINDER_AT_COL`.
2. Сохраняет новый `chat_id` и `message_id` в колонки 20 и 21.

Это влияет на следующий запуск напоминаний: счет не будет повторяться ежедневно, а дождется следующего понедельника.

## 12. HTTP endpoints

Внутри `bot.py` есть простой HTTP-сервер.

| Метод и путь | Что делает |
| --- | --- |
| `GET /` | Возвращает `OK`. Нужен для Render/UptimeRobot. |
| `HEAD /` | Возвращает `200` без тела. Сейчас это нормальный ответ на пинг UptimeRobot. |
| `GET /health` | Возвращает `OK`. |
| `HEAD /health` | Возвращает `200` без тела. |
| `GET /miniapp` | Отдает `miniapp.html`. |
| `HEAD /miniapp` | Возвращает `200` без тела. |
| `POST /api/requests` | Создает счет из Mini App. |
| `GET /migration` | Запускает dry-run или run миграции, если задан правильный `MIGRATION_SECRET`. |

Все остальные пути возвращают `404`.

Для UptimeRobot нормально видеть в логах строки вида:

```text
web: "HEAD / HTTP/1.1" 200 -
```

Раньше мог быть `501`, потому что `HEAD` не был обработан. Сейчас `200` - ожидаемое поведение.

## 13. Telegram Mini App и кнопка `Открыть`

Если задан `WEBAPP_URL`, при старте бот вызывает `set_chat_menu_button` и пытается поставить глобальную кнопку меню Telegram:

```text
Открыть -> WEBAPP_URL
```

Также `/start` в личном чате показывает inline-кнопку `Открыть форму`.

Важно:

- настройка кнопки меню зависит от Telegram/BotFather и клиента Telegram;
- `WEBAPP_URL` должен быть HTTPS;
- если `WEBAPP_URL` не задан, бот работает без Mini App и пишет warning в лог.

## 14. Миграция активных счетов на нового бота

Скрипт `migrate_active_invoices.py` нужен, когда меняется Telegram-бот и нужно переотправить активные счета от имени нового бота.

### 14.1. Почему миграция нужна

Сообщения и inline-кнопки принадлежат конкретному Telegram-боту.

Если поменять `BOT_TOKEN` на нового бота:

- новый бот не сможет обработать нажатия по старым сообщениям старого бота;
- старые `file_id` могут быть недоступны новому боту;
- активные счета лучше переотправить от имени нового бота.

### 14.2. Какие счета мигрируются

Скрипт берет только счета со статусом `Согласован`.

Не трогает:

- `Оплачено`;
- `Отклонен`;
- `Отменен`;
- `На согласовании`.

### 14.3. Что делает dry-run

`--dry-run` только показывает кандидатов и пишет лог:

- сколько найдено активных счетов;
- по каким чатам;
- у каких есть файлы;
- у каких нет файлов.

Telegram-сообщения не отправляются, таблица `requests` не меняется.

### 14.4. Что делает run

`--run`:

1. Берет старый токен из `BOT_TOKEN`.
2. Берет новый токен из `NEW_BOT_TOKEN`.
3. Для каждого активного согласованного счета отправляет новое сообщение от нового бота.
4. Если у счета есть файл, пытается скачать его старым ботом и отправить новым.
5. Обновляет в `requests` колонки 20 и 21 на новое сообщение.
6. Если файл был переотправлен и Telegram дал новый `file_id`, обновляет `FILE_ID_COL`.
7. Пытается удалить старое сообщение старого бота, если не включен `--keep-old`.
8. Пишет результат в лист `logs`.

### 14.5. Browser endpoint миграции

Если Render Shell недоступен, миграцию можно запустить через браузер:

```text
/migration?secret=<MIGRATION_SECRET>&mode=dry-run
/migration?secret=<MIGRATION_SECRET>&mode=run&confirm=RUN
```

Можно ограничить конкретным счетом:

```text
/migration?secret=<MIGRATION_SECRET>&mode=run&request_id=123&confirm=RUN
```

После завершения миграции `MIGRATION_SECRET` и `NEW_BOT_TOKEN` лучше удалить из Render, если они больше не нужны.

## 15. Callback-контракты

Эти строки важны: если поменять формат callback, нужно менять обработчик `button`.

| Callback | Где создается | Что значит |
| --- | --- | --- |
| `approve_<request_id>` | Кнопка согласования | Одобрить счет. |
| `reject_<request_id>` | Кнопка согласования | Отклонить счет и запросить причину. |
| `cancel_<request_id>` | Кнопка согласованного счета | Отменить уже согласованный счет и запросить причину. |
| `paid_<request_id>` | Кнопка согласованного счета | Начать ожидание чека от оплатчика. |
| `received_yes_<request_id>` | Кнопка у автора счета | Автор подтвердил, что оплата получена. |
| `received_no_<request_id>` | Кнопка у автора счета | Автор не подтвердил оплату, чек возвращается в рабочий чат. |
| `expense_team` | Выбор статьи расхода | Сохранить `Команда`. |
| `expense_ads` | Выбор статьи расхода | Сохранить `Рекламный бюджет`. |
| `expense_services` | Выбор статьи расхода | Сохранить `Сервисы`. |
| `skip_file` | Шаг файла в `/new` | Создать счет без файла. |
## 16. Временные состояния в памяти

В `bot.py` есть три словаря состояния. Они живут только в памяти процесса Render.

| Словарь | Ключ | Когда используется |
| --- | --- | --- |
| `user_state` | private chat ID | Пошаговое создание счета через `/new`. |
| `payment_state` | user ID оплатчика | Ожидание чека после кнопки `Оплатил`. |
| `reject_state` | user ID согласующего/отменяющего | Ожидание причины отклонения или отмены. |

Если Render перезапустится, эти временные состояния потеряются.

Примеры последствий:

- пользователь начал `/new`, Render перезапустился, нужно начать заново;
- оплатчик нажал `Оплатил`, Render перезапустился до отправки чека, бот уже не будет ждать чек;
- согласующий нажал `Отклонить`, Render перезапустился до ввода причины, причина уже не будет привязана к счету.

Основные данные счетов хранятся в Google Sheets и после перезапуска не теряются.

## 17. Права Telegram и ограничения

Для корректной работы бот должен:

1. Быть добавлен в нужные рабочие чаты.
2. Иметь возможность отправлять сообщения в эти чаты.
3. Иметь права удалять сообщения, если нужно удалять старые дубли и сообщения с чеками.
4. Получать сообщения и callback-кнопки.
5. Иметь личный `/start` от пользователей, которым он пишет в личку.

Если пользователь никогда не нажимал `/start` в личном чате с ботом, бот может не смочь отправить ему личное сообщение.

Если у бота нет прав удалить сообщение, бизнес-логика продолжит работу, но старые сообщения останутся в чате.

## 18. Что влияет на что

### 18.1. Статья расхода

Обычно влияет только на понедельничную группировку еженедельных напоминаний по оплате.

Исключение: для проектов `OR`, `ORKG`, `ОР`, `ОР КГ` статья `Рекламный бюджет` также меняет тег оплатчика на `@bulat_sufyanov`.

Не влияет на:

- согласующего;
- текст исходного счета;
- статус;
- чек.

### 18.2. Проект

Влияет на:

- какой чат получает счет;
- какой тег оплатчика будет в одобренном счете;
- какой проект увидит автор в сообщении о чеке;
- куда будут отправляться напоминания.

Если проект переименовать в листе `projects`, старые строки в `requests` не переименуются автоматически.

Для OR/ORKG есть отдельное правило: если проект нормализуется как `or`, `or kg` или `orkg` и статья расхода равна `Рекламный бюджет`, тег оплатчика выводится как `@bulat_sufyanov`. Это работает и для латиницы (`OR`, `ORKG`), и для кириллицы (`ОР`, `ОР КГ`).

### 18.3. Статус

Влияет на:

- попадание в напоминания;
- доступность дальнейшего сценария;
- миграцию активных счетов.

Напоминания работают только со статусами `На согласовании` и `Согласован`.

Миграция работает только со статусом `Согласован`.

### 18.4. `APPROVED_AT_COL`

Влияет на первое напоминание по оплате.

Если дата одобрения пустая у счета со статусом `Согласован`, первое напоминание по оплате может не сработать.

### 18.5. `LAST_PAYMENT_REMINDER_AT_COL`

Влияет на частоту напоминаний.

Если колонка пустая, бот может отправить первое напоминание на следующий день.

Если в колонке есть дата, бот будет ждать следующего понедельника.

Эта колонка используется и для счетов на согласование, и для счетов на оплату.

### 18.6. `LAST_INVOICE_MESSAGE_CHAT_ID_COL` и `LAST_INVOICE_MESSAGE_ID_COL`

Влияют на удаление предыдущего дубля счета.

Если там пусто или указан старый недоступный message ID, бот просто отправит новый счет, а старый не удалит.

### 18.7. `file_id`

Влияет на отправку счета и чека.

Telegram `file_id` связан с ботом. При смене Telegram-бота старые `file_id` могут не работать для нового бота. Для этого существует миграция.

## 19. Безопасная схема внесения правок

Перед изменением логики нужно проверить:

1. Не меняется ли порядок колонок в `requests`.
2. Не меняются ли названия статусов без обновления всех проверок в коде.
3. Не меняется ли формат callback без обновления обработчика `button`.
4. Не ломается ли общий путь Mini App и `/new`: оба должны создавать одинаковую строку в `requests`.
5. Не забыта ли очистка или обновление `LAST_PAYMENT_REMINDER_AT_COL` при смене статуса.
6. Не забыто ли сохранить новое сообщение в колонки 20 и 21 после отправки счета.
7. Не требует ли новая логика прав администратора в чате.
8. Не нужно ли обновить `MIGRATION.md`, если меняется миграция.

Минимальная техническая проверка после правок:

```bash
python -m py_compile bot.py migrate_active_invoices.py
```

Минимальная ручная проверка:

1. Создать тестовый счет через `/new`.
2. Проверить строку в `requests`.
3. Одобрить счет.
4. Нажать `Оплатил`.
5. Приложить чек.
6. Проверить сообщение автору счета.
7. Нажать `Да`.
8. Повторить отдельный тест с `Нет`.
9. Проверить отклонение и отмену с причиной.
10. Для напоминаний проверять на тестовой строке с датами и статусом, а не на боевых старых счетах.

## 21. Известные ограничения текущей архитектуры

1. Google Sheets используется как база данных, поэтому операции не атомарные. При одновременных действиях теоретически возможны гонки.
2. Номер счета считается через `len(sheet.get_all_values())`. Если вручную удалять строки, можно получить неожиданные номера.
3. Временные состояния живут в памяти Render и теряются при перезапуске.
4. Telegram может запретить удаление старых сообщений.
5. Для личных сообщений автору счета нужен предварительный `/start` в боте.
6. При смене `BOT_TOKEN` старые сообщения и `file_id` требуют миграции.
7. Старые строки без заполненных колонок 20 и 21 не всегда можно очистить от дублей автоматически.
8. Статья расхода не является отдельным справочником в Google Sheets, она зашита в коде и Mini App.

## 22. Краткая карта процесса

```text
/new или Mini App
  -> проект найден в projects?
      нет -> просим ввести проект заново / возвращаем ошибку Mini App
      да
        -> выбор статьи расхода
        -> кому платим
        -> сумма
        -> файл или пропуск
        -> комментарий
        -> строка requests: На согласовании
        -> сообщение в чат проекта: Одобрить / Отклонить

Одобрить
  -> статус Согласован
  -> approved_at = сегодня
  -> last_reminder = пусто
  -> удалить сообщение на согласование
  -> отправить счет на оплату: Оплатил / Отменить счет

Отклонить
  -> запрос причины
  -> статус Отклонен
  -> удалить сообщение счета, если можно
  -> автору: счет не согласован, причина

Отменить счет
  -> запрос причины
  -> статус Отменен
  -> удалить сообщение счета, если можно
  -> автору: счет отменен, причина

Оплатил
  -> бот ждет чек от этого user_id
  -> чек получен
  -> статус Оплачено
  -> сохранить данные чека
  -> удалить сообщение пользователя с чеком, если можно
  -> отправить чек в рабочий чат reply к счету
  -> отправить чек автору: Оплата получена? Да / Нет

Автор нажал Да
  -> убрать кнопки
  -> статус остается Оплачено
  -> написать автору: Напиши /new чтобы отправить счет

Автор нажал Нет
  -> убрать кнопки
  -> статус Согласован
  -> last_reminder = сегодня
  -> вернуть чек в рабочий чат с текстом, что оплата не получена
  -> снова показать кнопку Оплатил

Ежедневный job напоминаний
  -> найти На согласовании и Согласован
  -> если первое напоминание: следующий день
  -> если повторное: понедельник
  -> удалить предыдущий дубль, если можно
  -> переотправить счет
  -> обновить last_reminder и last message ids
```
